# Микросервис для выдачи промокодов

1. Введение и область применения

Введение: настоящий документ описывает требования к проектированию и разработке микросервиса для выдачи промокодов, предназначенного для поддержки программ лояльности клиентов и автоматизации маркетинговых акций.

Область применения: микросервис предназначен для хранения и выдачи промокодов пользователям в рамках маркетинговых акций. Микросервис автоматически проверяет право пользователя на получение промокода и предотвращает его повторную выдачу в рамках одной акции.

2. Цели и задачи  
Цели:
- предотвращение повторной выдачи промокода одному пользователю в рамках одной акции;
- автоматизация проверки права пользователя на получение промокода;
- снижение нагрузки на основную платформу в период проведения маркетинговых акций.  
Задачи  
- спроектировать микросервис со своей собственной базой данных;
- разработать структуру реляционной базы данных для хранения пользователей, акций, промокодов и истории выдач;
- реализовать автоматическую проверку права пользователя на получение промокода;
- обеспечить сохранение информации о каждой успешной выдаче промокода.
  
3. Требования

3.1 Функциональные требования
- система должна автоматически проверять статус акции при каждом запросе на получение промокода;
- система должна определять, выдавался ли ранее промокод пользователю в рамках данной акции;
- cистема должна блокировать повторную выдачу промокода одному пользователю в рамках одной акции;
- при успешной проверке система должна выбрать свободный промокод из базы данных;
- после успешной выдачи система должна сохранить информацию о выдаче в таблице issuances, включая:
идентификатор пользователя (user_id);
идентификатор акции (action_id);
идентификатор промокода (promocode_id);
дату и время выдачи;
- после успешного сохранения записи система должна вернуть пользователю промокод.
  
3.2 Нефункциональные требования
- время проверки права пользователя на получение промокода не должно превышать 50 мс;
- время обработки запроса на выдачу промокода не должно превышать 200 мс;
- не менее 95% успешных запросов должны выполняться за время не более 200 мс;
- сервис должен обеспечивать доступность не ниже 99% времени в течение месяца;
- сервис должен выдерживать нагрузку 500 запросов в секунду.
